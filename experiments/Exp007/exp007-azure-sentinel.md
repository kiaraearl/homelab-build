## Objective
Deploy Microsoft Sentinel as a cloud SIEM, connect an on-premises Windows Server Domain Controller via Azure Arc, ship Windows Security Event logs to the cloud, write KQL detection queries, and build an automated brute force detection rule that fires real incidents.

---

## Why This Matters
Most enterprise SOC environments run on Microsoft's security stack. Having hands-on experience with Sentinel — not just Splunk — makes you significantly more competitive at Tier 1. This experiment proves you can:

- Deploy and configure a cloud-native SIEM from scratch
- Connect on-premises infrastructure to Azure using Arc (hybrid cloud)
- Write KQL queries to hunt for threats in real log data
- Build custom analytics rules mapped to MITRE ATT&CK
- Triage a real incident in a SOC queue

## What I Did

### Step 1 — Created Azure Free Account
Signed up for a free Azure account at azure.microsoft.com/free. Microsoft provides $200 in credits for 30 days plus always-free services. Verified identity and confirmed subscription (Azure subscription 1) with Owner role.

### Step 2 — Created Log Analytics Workspace
Navigated to Log Analytics Workspaces → Create. Deployed **LAW-SOCLab** in resource group **RG-SOCLab** in East US. The LAW is the backend storage engine — Sentinel sits on top of it and all logs are queried from here using KQL. This distinction (LAW = storage, Sentinel = analytics layer) is a common interview question.

### Step 3 — Deployed Microsoft Sentinel
Navigated to Microsoft Sentinel → Create → selected LAW-SOCLab. Sentinel free trial activated (10 GB/day free through 7/1/2026). Sentinel is now the SIEM layer — detection rules, incidents, dashboards, and threat hunting all live here.

### Step 4 — Onboarded WinServer2022-DC01 via Azure Arc
WinServer2022-DC01 is a local VirtualBox VM, not an Azure VM, so it can't be managed by Azure directly. Azure Arc solves this — it registers on-premises and local machines with Azure so they appear as managed resources.

**Setup steps:**
- Installed VirtualBox Guest Additions on WinServer2022-DC01 to enable file transfer
- Set up shared folder between Windows 11 host (`C:\VMShare`) and WinServer2022-DC01 (auto-mounted as Z: drive)
- Generated Arc onboarding script from Azure portal (Azure Arc → Machines → Onboard existing machines → Windows, East US, no proxy)
- Copied script to `C:\VMShare` on host → accessed at `Z:\` inside VM
- Ran script in PowerShell as Administrator on WinServer2022-DC01 — browser authentication prompt appeared, signed in with Azure account
- Script registered DC01 as an Arc-enabled server in RG-SOCLab

**Result:** DC01 appeared in Azure Arc → Machines with status **Connected**.

### Step 5 — Deployed Azure Monitor Agent to DC01
Navigated to Azure Arc → Machines → DC01 → Extensions → Add → Azure Monitor Agent. Extension deployed successfully (version 1.42.0.0, status: Succeeded).

Created a **Data Collection Rule** via Sentinel → Configuration → Data connectors → Windows Security Events via AMA → Open connector page → Create data collection rule:
- Rule name: `DCR-WinServer-SecurityEvents`
- Resource: DC01
- Collect: All Security Events

Windows Security Event logs began flowing from DC01 to LAW-SOCLab.

### Step 6 — Verified Log Ingestion with KQL
Opened Sentinel → Logs and confirmed data flowing:

```kql
SecurityEvent
| take 10
```

Returned live Security Events from DC01.lab.local — timestamps, accounts, event source confirmed.

### Step 7 — Wrote KQL Detection Queries
**Query 1 — Failed Logon Hunt (Event ID 4625):**
```kql
SecurityEvent
| where EventID == 4625
| project TimeGenerated, Account, Computer, EventID, Activity
| order by TimeGenerated desc
```
Returned 5 failed logon attempts against LAB.LOCAL\Administrator — events generated by simulated brute force script.

**Query 2 — Brute Force Summary:**
```kql
SecurityEvent
| where EventID == 4625
| summarize FailedAttempts = count() by Account, Computer
| order by FailedAttempts desc
```
Result: `LAB.LOCAL\Administrator | DC01.lab.local | 5` — clean brute force pattern in one row.

Both queries saved to Sentinel Logs.

### Step 8 — Created Brute Force Analytics Rule
Navigated to Sentinel → Configuration → Analytics → Create → Scheduled query rule.

**Rule configuration:**
- Name: `Brute Force - Multiple Failed Logons`
- Description: Detects 5 or more failed logon attempts against the same account within 5 minutes
- Severity: **High**
- MITRE ATT&CK: **Credential Access → T1110 (Brute Force)**
- Status: Enabled

**Rule query:**
```kql
SecurityEvent
| where EventID == 4625
| summarize FailedAttempts = count() by Account, Computer, bin(TimeGenerated, 5m)
| where FailedAttempts >= 5
```

- Query runs every: 5 minutes
- Lookup window: 5 minutes
- Incident creation: Enabled

### Step 9 — Incident Fired
Generated 6 failed logon attempts on WinServer2022-DC01 using `Z:\failedlogins.ps1`. Within 5 minutes, Sentinel fired **Incident #1 — Brute Force - Multiple Failed Logons** with:
- Severity: High
- Status: New
- Alerts: 1
- Events: 1
- Description: "Detects 5 or more failed logon attempts against the same account within 5 minutes"

## Troubleshooting Notes
- **Azure Monitor Agent via Arc extension** reported "Succeeded" in portal but service did not register on VM — this is a known VirtualBox/NAT limitation with Arc extension deployment timing. Resolved by waiting for agent heartbeat and verifying via `azcmagent check`.
- **Drag and drop to VM** failed initially — fixed by installing VirtualBox Guest Additions and enabling Bidirectional under Devices → Drag and Drop.
- **PowerShell script execution** blocked by execution policy — resolved with `Set-ExecutionPolicy -ExecutionPolicy Bypass -Scope Process` before running scripts from shared folder.
- **aka.ms/AzureMonitorAgentWindowsInstaller** redirected to Bing — manual MSI install not supported on Windows Server; correct method is Arc extension deployment.

## Key Takeaways
- **Sentinel ≠ Log Analytics Workspace** — LAW stores the data, Sentinel provides the analytics layer on top. Two separate services.
- **Azure Arc** is how enterprises manage on-premises and hybrid infrastructure from Azure. Real-world SOC environments use this constantly.
- **KQL** is the query language for Sentinel, Microsoft Defender, and Azure Monitor — investing in KQL pays off across the entire Microsoft security stack.
- **MITRE ATT&CK mapping** on analytics rules is standard practice in enterprise SOC environments — every detection should be mapped to a tactic and technique.
- **Shared folders** between host and VM are a fundamental IT support skill — essential for file transfer when network or clipboard access is limited.
